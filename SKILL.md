# Prokopē — engineering long goal loops

Named for Greek **προκοπή** (*prokopē*): progress as continuous advancement — the climb, not the arrival. The loop is for the road.

A goal loop is the architecture of pursuit under discontinuity: sessions end, context windows fill, models change, and the goal must survive all of it. This skill is how to write a goal that a fresh session can pick up cold, how to keep both honesty and search pressure alive across days of autonomous work, and how to recognize the failure modes long goals fall into.

## §0 · Self-updating

Keep this skill true when the underlying fact moves:

- **A new collapse mode is exhibited on a long goal** → add it to the §3 catalog; if it generalizes, add a refuse rule to the brief template.
- **A companion contract changes** (sortes / fabrica / ISDA) → update §5.
- **The session re-arm or compactification protocol changes** → update §4, §7, the templates, and `references/compactification-survival.md` together.
- **The tip or orientation-packet shape changes** → update §6–§7 and the goal-brief template.

Live campaign state belongs in the goal's own working surfaces (search log, status ledger) — never in this file.

## §1 · Identity

**You author and maintain goal loops** — not the domain work inside them. The loop's enemy is bearer discontinuity in every form: sessions end, models change, and context windows compact mid-flight.

| This skill **is** | This skill **is not** |
|-------------------|------------------------|
| How to write a goal brief that survives 24+ hours **and** a mid-session compaction | A substitute for domain skill (math, code, ops) |
| How to structure working surfaces (search log, ledger, notes) so tips outlive context | fabrica (no agent dispatch) |
| How to keep honesty and pressure both alive across boundaries | A checklist that completes the goal |
| How to wire sortes / ISDA / fabrica into the loop | A universal wave-count or hour-count quota |
| Compactification-survival methodology (pre/post boundary protocols) | A pre-compaction hook replacement (hooks are advisory) |

**One-line law:** true success is named and hard; process volume is never success; state lives in **tips on disk**, not in the context window; when pressure dies or a boundary corrupts continuity, re-arm — do not add more iterations in the same dead or ghost basin.

## §1.1 · Goal classes (scope the doctrine)

Not every multi-session goal has the same shape. **Name the class in the brief.**

| Class | Shape | Sortes floor | Compactification package | Honest-negative product |
|-------|-------|--------------|--------------------------|-------------------------|
| **A** Open constructive (research until true success) | Full search log + pressure package | Mandatory multi-arm draws after kickoff | **Required** (full §7) | Forbidden as completion |
| **B** Assay / survey | Bounded questions; time/scope exit allowed | Multi-arm draws recommended | Required light (orientation packet + tip write) | Allowed once the questions are answered |
| **C** Ship / instrument | Deliverable + tests | Review catalog if code is multi-arm | Tip = branch/PR + test bar; light packet | Not applicable |
| **D** Ordinary multi-session | Light surfaces | Optional | Optional tips; still refuse summary-as-warrant | Optional |

Class-A doctrine is the strictest; do not apply its refuse lists to class B/C goals by default.

## §2 · Dual success modes (non-negotiable)

Every goal brief must separate two things that long campaigns constantly conflate:

| Mode | Meaning | Completes the goal? |
|------|---------|---------------------|
| **True success** | The domain predicate the goal exists for (theorem, working system, proof, deploy) | **Yes** — only this, or an explicit stop by the person who owns the goal |
| **Continuable handoff** | Surfaces updated, tip ordered, pressure noted, goal still open | **No** — this is survival across sessions **and** compaction boundaries, not completion |

**Refuse as "done":**

- Minimum-N iterations, pipelines, or reports
- "Honest negative" shipped as the product when the brief forbids early close
- A "substantial progress" checklist with the true-success bar unmet
- Intermediate results elevated by vocabulary change alone
- **Summary or compaction residue treated as completion evidence**

A loop can stay perfectly honest — every status entry scoped and true, the target never oversold — while its falsification rate quietly collapses toward zero. **Honesty is not altitude.** And a loop can stay honest while its continuity silently corrupts at a boundary — tips lagging while the conversation assumes they are current. **Honesty is not compact survival either.** Instrument all three separately (§4, §7).

## §3 · Collapse catalog

Diagnostic lenses for a long goal that is failing. Each names a way long goals actually fail; treat them as diagnostics for a running loop and a refuse-list for the brief's author. Full list with one-liners in `references/collapse-catalog.md`.

| Mode | Symptom | Prescription |
|------|---------|--------------|
| **Count-collapse** | Iteration/commit count treated as progress | The bar is true success only; handoff ≠ done |
| **Volume ≠ altitude** | Huge search log, lateral re-skins | Density checkpoints on deltas; change the engine |
| **Kill-pressure death** | Falsification rate → 0; late work never dies | Pressure health (§4); re-arm the session |
| **False decorrelation** | Draw names vary; outcome skeleton freezes | Outcome-shape check; force a falsifying arm |
| **One-engine costume** | One generative model re-skinned across vocabularies | Name the engine class; require a zero-entry class next |
| **Adversary sleep** | No adversarial review after the early hours | Mandatory adversarial cadence in the brief |
| **Trail lapse** | Draws untagged; no `not surfaced:` trails | Hygiene fail; re-arm; draw unworked lenses |
| **Elevation by packaging** | Substrate renamed as a summit claim | Standing blocks + claim-scale language |
| **Tip lag** | Status ledger behind the search log | Resume from the last continuable tip, never a mid-file head |
| **Overfit gospel** | A prior campaign's constants treated as universal law | Transfer modes, not numbers |
| **Summary-as-warrant** | Compaction/summary text treated as exhibit for bars or claims | Re-read tips; the summary is forensics only (§7) |
| **Ghost continuity** | Standing blocks, draws, or next moves assumed still held in chat after a boundary | Orientation packet; re-draw sortes |
| **Tip-erasure** | A boundary accepted without a tip write | Pre-boundary protocol is mandatory |
| **Re-constitution from summary** | Doctrine or the bar rewritten from blurred memory | The brief is law; never re-author it under compaction fog |
| **Soft-elevation after compaction** | Summary "progress" read as altitude | Dual success modes; only tips elevate |

**Refuse-list for this skill itself:** do not turn any prior campaign's brief into the template with the nouns sanded off; do not hardcode restart cadences as universal constants; do not clone the orchestration companion; do not treat advisory pre-compaction hooks as a substitute for tip-write discipline.

Full mode list: `references/collapse-catalog.md`. The compactification family in detail: `references/compactification-survival.md`.

## §4 · Pressure health (required package for class A)

Honesty can hold while search dies. Instrument both. Compaction boundaries often **mask** pressure death — the summary still sounds productive.

### Minimal signals (per session or per ~N iterations)

| Signal | Healthy | Sick |
|--------|---------|------|
| **Live falsifications** | ≥1 real kill or demotion of a *load-bearing* claim recently | Zero live kills across a long tail |
| **Draw hygiene** | Draws tagged; `not surfaced:` trails present on multi-arm waves | Untagged multi-arm work; trail silence |
| **Outcome diversity** | The draw changes the *shape* of the work, not just its titles | A fixed template regardless of draw |
| **Adversarial touch** | A recent external attack on the elevation path | Only self-review |
| **Engine-class ledger** | New attempts name their generative engine class | Same engine, new nouns |
| **Tip freshness** | Search-log tip and status tip agree; written before the last boundary | Tip lag or tip-erasure after a compaction |

### The re-arm rule (default operational fix)

When pressure is sick — or context depth is high in a long marathon — or post-boundary orientation fails (§7):

1. **Checkpoint** — update the search-log tip, the status ledger's continuable tip, and a pressure line; commit and push.
2. **End the session.** Do not keep pumping iterations in the same saturated or ghost-continuous context.
3. **Re-post the goal brief** into a fresh session (same brief file — it is designed for this).
4. A fresh context restores kill pressure, tag discipline, and genuine draw-following **if** the surfaces carry the state.

Cadence is heuristic and domain-local. Prefer **pressure-signal and tip-integrity triggers** over fixed clocks, and let the goal's owner set thresholds in the brief.

**Anti-thrash:** re-arm is not free. If two consecutive sessions re-arm without a load-bearing kill *or* an engine-class change, stop pumping — write a plateau note and change the generative engine before the next re-post. The count of re-arms is not progress either.

**Governance (class A):** the brief should say who may pause the goal, who re-posts after unattended runs, and which calls belong to the goal's owner rather than the agent.

### Sortes doctrine (goal-loop floor)

Paths shown for the Claude Code layout — substitute your harness's skills directory (e.g. `.grok/skills/`) once and these hold everywhere.

```bash
# Research / theory / design campaigns (default for long constructive goals):
bun .claude/skills/sortes/bin/pick.ts --catalog .claude/skills/sortes/lenses-ideation --n <arms>

bun .claude/skills/sortes/bin/pick.ts --n <arms>
```

- Tag every multi-arm wave with the drawn lens names.
- End consumer artifacts with a `not surfaced:` line naming the angles the draw did not reach.
- If tags lapse for a long stretch → hygiene fail → re-arm the session; do not push through.
- **Outcome-shape check:** if the wave skeleton is identical across different draws, decorrelation has failed — force a falsifying arm or change the engine class.

### Fabrica (inside the goal)

Use it for wide analysis and multi-basin surveys *inside* the goal. Dual-write and manifests are its contract; prokopē owns what lands on which surface (§6). Never treat "the pipeline ran" as altitude. Dispatched agents never had the transcript — dual-write is *their* compactification organ; the goal's tips remain the campaign's.

## §6 · Working surfaces (goal-loop state architecture)

Surfaces exist so compaction and session death are survivable — not as documentation theater.

| Surface | Role | Failure risk |
|---------|------|--------------|
| **Goal brief / task record** | The law: true success, process constitution, refuse list, orientation packet | Stale doctrine if not updated in place |
| **Search log** (append-only) | System of record for attempts; its tip is the continuable state | Node-count cosplay |
| **Status ledger** | Elevation registry + standing blocks + the open tip | Tip lag behind the search log |
| **Research notes / intermediate claims** | Soft consolidation with explicit non-claim banners | "Soft" drifting into "win" |
| **Executable exhibits** (code, formal proofs, tests) | Typed truth; elevation gates encoded in the artifact | Volume cosplay |
| **Version control tip** | Landable continuity | Merged ≠ succeeded |

**Promotion flow:** exhibit → search-log node → (optionally) research note → status ledger only if the elevation bar is met — and never in reverse; the ledger cannot invent altitude.

**Resume rule:** always resume from the last continuable tip plus the search-log tip — never from a mid-file status head that may lag, and never from a compaction summary.

**Tip contract (every surface):** one named tip; the tip write is atomic and happens *before* any boundary; resume reads tips only; tip lag is a hygiene fail to reconcile before new work.

## §7 · Compactification survival methodology

This is a first-class operation-space, not a footnote to re-arm. Long goals fail at the boundary itself when the doctrine lives in chat and dies at a compaction or session cut. Full reference: `references/compactification-survival.md`.

### Two boundaries

| Boundary | What happens | Continuity organ |
|----------|--------------|------------------|
| **Soft compaction** | Mid-session summarization; the same session continues with degraded memory | Disk tips + the brief (already written) |
| **Hard re-arm** | The session ends; a fresh session re-posts the goal brief | The same surfaces + a cold brief re-read |

Harness pre-compaction hooks are **advisory** — they warn, they do not write tips. This methodology owns the operational write.

### One law

**Anything load-bearing after a boundary must already be on disk in tip-form before the boundary — or it is already lost.**

Context is scratch. Surfaces are the continuity organ. **The summary is not a surface.**

### What must never live only in chat

The true-success bar · standing blocks and the refuse list · the search-log tip and status tip · ordered next moves · recent load-bearing kills · the engine class of the current attempt · in-flight draw tags and `not surfaced:` trails · owner decisions that change the bar.

### Orientation packet (cold re-entry order)

Every class-A (and light class-B) brief names an **ordered re-read set**:

```text
1. Goal brief (law + true success + refuse list)
2. Status ledger tip (elevation + open bar + standing blocks)
3. Search-log tip (frontier + ordered next moves)
4. Exhibit paths cited by the tips only
5. Last pressure note (if kept separately)
```

**Tips first.** Never open with "read the whole log." Expand on demand.

### Pre-boundary protocol (before a compaction or a pause)

1. Search-log tip: last entries + **ordered next moves**.
2. Status tip: bar still open (unless truly elevated); standing blocks intact.
3. Pressure line: last live kill, or an explicit "pressure sick → re-arm."
4. Land durable work in version control.
5. Only then accept the compaction or end the session.

**Refuse:** "I'll remember" or "the summary will keep it" without a tip write.

### Post-boundary protocol (after a compaction or a cold start)

1. Run the orientation packet **from disk**.
2. **Refuse summary-as-warrant** for: the bar's status, standing blocks, tip identity, the kill ledger, owner decisions.
3. If tips lag, pressure is sick, or the standing blocks are uncertain → **hard re-arm**; do not soft-continue into more work.
4. Re-draw sortes for the next multi-arm wave — never inherit "already drew" from a summary.
5. Resume from the **ordered next moves** on the tip, not from summary "progress so far."

### Soft continue vs hard re-arm

| Signal | Prefer |
|--------|--------|
| Tips written, pressure healthy, one compaction, brief still loadable | Soft continue, **after** the post-boundary re-read |
| Tips stale or lagging; pressure sick; standing blocks uncertain; multiple compactions; marathon depth | **Hard re-arm** |
| The summary claims altitude the tips do not | Hard re-arm + a plateau note |
| Two soft-continues after compaction without a load-bearing kill or an engine change | Hard re-arm + change the engine before re-posting |

### Brief design tests (compactification package)

A class-A brief fails this surface if it lacks: an ordered orientation packet · a pre-boundary tip-write rule · a post-boundary "re-read tips; refuse summary warrant" rule · the bar and refuse list in the brief file rather than chat lore · resume language pointing at tips rather than "the conversation so far" · boundary-triggered re-arm criteria.

## §8 · Authoring a goal brief (procedure)

1. **Name true success** in one falsifiable sentence (a domain predicate).
2. **Name the failure modes of prior attempts** — an explicit do-not-repeat list.
3. **Obstacle protocol** — name the block → fan out ≥3 attack vectors → ≥1 that rewrites a foundation → ≥1 deliberately high-risk path with a kill criterion → experiment → loop. Adapt counts to the domain; keep the shape.
4. **Honesty package** — standing blocks; "what is not success"; claim-scale language rules.
5. **Pressure package** — falsification cadence; adversarial-review cadence; sortes rules; re-arm triggers.
6. **Surface contracts** — which files are the system of record for search vs elevation; **named tips**.
7. **Compactification package** — the orientation packet (ordered paths); the pre-boundary tip-write rule; the post-boundary refuse-summary-as-warrant rule; soft-continue vs hard re-arm criteria.
8. **Companion block** — paste the sortes/fabrica/ISDA invocations into the goal prompt.
9. **C0–C1** — telos and substrate explicit; run C5 on the brief itself (a few hundred characters must still preserve "true success vs handoff ≠ done").
10. **Handoff ≠ done** — a session end or a soft compaction produces continuable state, nothing more.
11. **Adversarial pass on the brief itself** — does any checklist item quietly become an exit ticket? Does any step assume continuous memory across a compaction?

Template: `templates/goal-brief.md`. Paste block: `templates/goal-prompt-block.md`.

## §9 · Operating a live goal (procedure)

1. Orient: run the **orientation packet** (brief → status tip → search-log tip → cited exhibits → pressure note).
2. Draw sortes (correct catalog); inject the lenses; plan the arms.
3. Execute experiments; write exhibits first.
4. Log search-log nodes with tags, kills, fan-outs, and `not surfaced:` trails.
5. Update the status ledger only for elevation-relevant truth.
6. **Pressure check** at every re-arm window, or when sick signals appear.
7. **Before any compaction, pause, or session end:** run the pre-boundary tip write (§7).
8. **After any compaction or cold start:** run the post-boundary re-read; refuse summary warrant; hard re-arm if continuity failed.
9. Checkpoint version control; if sick → end the session and re-post the brief.
10. Never complete the goal on process volume or summary residue.

## §10 · Acceptance tests (class-parameterized)

**All classes:**

- [ ] True success is one sentence and falsifiable
- [ ] Continuable handoff explicitly ≠ completion
- [ ] Refuse list includes count-collapse, packaging elevation, and **summary-as-warrant**
- [ ] No "minimum N iterations" as success
- [ ] Class named (A/B/C/D)
- [ ] Resume language points at **tips**, not mid-file heads or conversation memory

**Class A adds:** prior failure modes · obstacle/fan-out protocol · full pressure package · sortes multi-arm doctrine · search log + ledger + exhibits named · **the full compactification package** (orientation packet, pre/post boundary protocols).

**Class B:** bounded question list + a scope/time exit · survey surfaces named · light orientation packet + tip write.

**Class C:** deliverable + test bar · review-catalog sortes if code goes multi-arm · a branch/PR tip is fine.

**Class D:** light surfaces · sortes optional · still refuse summary-as-warrant when multi-turn.

A running class-A loop is healthy only if: a recent load-bearing falsification exists (or a named plateau plus an engine change) · multi-arm waves are tagged · tip sources agree · standing blocks are enforced · **the last boundary left the tips coherent** (no tip-erasure, no ghost continuity).

## §11 · One-line summary

**Write goals that name a hard true success, instrument honesty and kill-pressure as separate axes, force external decorrelation and telos discipline, carry state in tips on disk rather than context windows, survive soft compaction and hard re-arm with explicit pre- and post-boundary protocols, and never confuse iteration count or summary residue with altitude.**
