# Compactification survival — reference

Companion to prokope §7. Load when authoring class A/B goal briefs or diagnosing post-boundary drift.

## Why this exists

Long goals outlast a single coherent context. Two boundaries erase in-context state:

| Boundary | What happens | Continuity organ |
|----------|--------------|------------------|
| **Soft compaction** | Mid-session summarization; the same session continues with degraded memory | Disk tips + the brief (already written) |
| **Hard re-arm** | The session ends; a fresh session re-posts the goal brief | The same surfaces + a cold brief re-read |

The characteristic failure: a loop with honest claims and real surfaces still drifts at the boundary, because summary residue and mid-file "status heads" get treated as warrant while standing blocks and ordered next moves live half in chat. That is compactification failure — a distinct axis from kill-pressure death, and it needs its own protocols.

## One law

**Anything load-bearing after a boundary must already be on disk in tip-form before the boundary — or it is already lost.**

Context is scratch. Surfaces are the continuity organ. The summary is **not** a surface.

## What must never live only in chat

| Must survive on disk | Why |
|----------------------|-----|
| The true-success bar (one sentence) | Summaries soften "only completion bar" into "substantial progress" |
| Standing blocks / refuse list | Compaction forgets prohibitions first |
| Search-log tip + status open-tip (paths and content) | The resume source of truth |
| Ordered next moves | The default after a compaction is re-discovery theater |
| Recent load-bearing kills and demotions | Pressure dies silently if it lives only in the transcript |
| The engine class of the current attempt | Re-skinning after a compaction is free without a ledger |
| In-flight draw tags + `not surfaced:` trails | Trail lapse |
| Owner decisions that change the bar | "We agreed in chat" evaporates |

Fine to leave in chat: intermediate brainstorms, dead-end texture already logged as search-log nodes, raw tool dumps already dual-written.

## Tip contract (every working surface)

1. **One named tip** per surface (a marker, a heading, a ledger line).
2. **The tip write is atomic** — finish it *before* accepting a pause, a compaction, or a session end.
3. **Resume reads tips only** — never mid-file narrative, never the compaction summary.
4. **Tip lag is a hygiene fail** — if the status tip and the search-log tip disagree, reconcile before any new work.

## Orientation packet (cold re-entry order)

A compaction-surviving goal names an **ordered re-read set** in its brief. The class-A default:

```text
1. Goal brief (law + true success + refuse list)
2. Status ledger tip (elevation + open bar + standing blocks)
3. Search-log tip (frontier + ordered next moves)
4. Exhibit paths cited by the tips only
5. Last pressure note (kills / sick signals / engine class), if kept separately
```

Do **not** open with "read the whole log." Tips first; expand on demand.

## Pre-boundary protocol (before a soft compaction or a hard pause)

The minimum write set (adapt names to the domain):

1. Search-log tip: last entries + **ordered next moves**.
2. Status tip: the bar still open (or an elevated claim with its warrant); standing blocks intact.
3. Pressure line: the last live kill, or an explicit "pressure sick → re-arm."
4. Land durable work in version control.
5. Only then accept the compaction or end the session.

**Refuse:** "I'll remember" or "the summary will keep it" without a tip write.

## Post-boundary protocol (after a compaction or a cold start)

1. Run the orientation packet **from disk**, in order.
2. **Refuse summary-as-warrant** for: the bar's status, standing blocks, tip identity, the kill ledger, owner decisions.
3. If tips lag, pressure is sick, or the standing blocks are uncertain → **hard re-arm** (end the session, re-post the brief); do not soft-continue into more work.
4. Re-draw sortes for the next multi-arm wave — never inherit "already drew" from a summary.
5. Resume construction from the **ordered next moves** on the tip, not from summary "progress so far."

## Soft continue vs hard re-arm

| Signal | Prefer |
|--------|--------|
| Tips written, pressure healthy, one compaction, brief still loadable | Soft continue, after the post-boundary re-read |
| Tips stale or lagging; pressure sick; standing blocks uncertain; multiple compactions; marathon depth | **Hard re-arm** |
| The summary claims altitude the tips do not | Hard re-arm + a plateau note |
| Two soft-continues after compaction without a load-bearing kill or an engine change | Hard re-arm + change the engine before re-posting |

Hard re-arm is the default cure once a boundary has corrupted the in-context constitution. Soft continue is allowed only when the orientation packet *verifies* continuity.

## Brief design tests (compactification package)

A class-A brief fails compactification survival if:

- [ ] No ordered orientation-packet paths
- [ ] No pre-boundary tip-write rule
- [ ] No post-boundary "re-read tips; refuse summary warrant" rule
- [ ] The bar or refuse list lives only in chat lore, not in the brief file
- [ ] Resume language points at a "status head" or "the conversation so far" instead of tips
- [ ] Re-arm is described only as "when tired," with no boundary-triggered tip protocol

## Collapse modes (compactification family)

| Mode | Symptom | Prescription |
|------|---------|--------------|
| **Summary-as-warrant** | Compaction/summary text treated as exhibit for claims or bars | Re-read tips; the summary is forensics only |
| **Ghost continuity** | Standing blocks, draws, or next moves assumed still held in chat | Orientation packet; re-draw sortes |
| **Tip-erasure** | A boundary accepted without a tip write | Pre-boundary protocol is mandatory |
| **Re-constitution from summary** | Doctrine rewritten from blurred memory | The brief is law; never re-author the bar in compaction fog |
| **Soft-elevation after compaction** | Summary "progress" read as altitude | Dual success modes; only tips elevate |
| **Orientation inversion** | Reading the whole corpus before the tip on re-entry | Tips first; expand on demand |
| **Boundary thrash** | Compaction → soft continue → compaction, without a tip or a kill | Hard re-arm; anti-thrash with the pressure package |

## Harness notes

- Pre-compaction hooks, where a harness offers them, are **advisory** — they warn; they do not write tips. This methodology owns the operational write.
- Session-start injection (rules files, system prompts) restores *doctrine*, not *goal tips* — the tips still come from the surfaces.
- Dispatched subagents never had the transcript — the dual-write contract (see the fabrica companion) is their version of this discipline; the goal's own tips remain the campaign's organ.
- A version-control push is landable continuity, never true success.
